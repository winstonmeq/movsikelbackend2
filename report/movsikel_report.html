1. Executive Summary

MovSikel is a fully working ride-hailing platform purpose-built for the tricycle transport market in Kidapawan City. The system is live in production on a dedicated cloud server (VPS) with a real-time matching engine, a driver wallet/commission system, push-notification dispatch, and live GPS tracking.

The platform is designed around how tricycles actually operate locally: passengers pay the driver in cash, the app is a digital matchmaker — not a payment processor — which keeps the model simple, compliant, and free of financial-handling risk while still generating platform revenue through a transparent booking-fee scheme.

Two ride products:
Shared Ride — a standard "pasada" pickup, free for the driver, no fare negotiation.
Booking Ride (Pakyaw) — a complete door-to-door route where the passenger offers a fare and the driver accepts or declines. This premium product carries the platform's booking fee.

3. Passenger–Driver Matchmaking

The core of the platform is a staged sequential dispatch engine that connects a passenger to the right driver quickly and fairly. The full flow:

Passenger requests a ride. They pick a pickup & destination (via Google Places), choose Shared or Booking, and — for Booking — offer a fare. The request is sent to the backend.
The system finds nearby drivers. A geospatial query locates online drivers near the pickup point, ordered nearest-first, and builds a dispatch queue.
First batch is offered. The ride is offered simultaneously to the first 3 nearest drivers via push notification — maximizing the chance of a fast accept.
First to accept wins. The first driver to tap "Accept" is assigned the ride atomically; the offer is withdrawn from the others instantly.
Auto-advance if no one accepts. If the batch doesn't respond within the offer window, the system advances to the next nearest driver, one at a time, until someone accepts.
Graceful exhaustion. If every nearby driver has been asked and none accept, the passenger is cleanly told "No drivers available" rather than waiting indefinitely.

3.1 Finding Drivers — Geospatial Search

Driver discovery uses MongoDB's 2dsphere geospatial index with a $near query. This is an indexed R-tree lookup — it does not scan every driver. It returns only drivers within the search radius, already sorted by true distance, in milliseconds. This keeps the system fast and inexpensive even as the driver pool grows.

3.2 Why This Design Is Strong

  • Fast: Offering 3 drivers at once dramatically reduces match time vs. asking one-by-one.
  • Fair: Only the truly nearest, currently-online drivers are offered each ride.
  • Race-proof: "First-to-accept-wins" is enforced atomically at the database level — two drivers can never be assigned the same ride.
  • Serverless-resilient: Dispatch progresses lazily on each poll/event, so it works reliably without fragile background timers.
  • Scalable foundation: The architecture is ready to layer in fairness-based driver scoring (idle time, ratings, acceptance rate) when the network grows.

4. Fare Scheme & Driver Wallet (Revenue Model)

Passengers pay cash directly to the driver — the platform never touches passenger money. Platform revenue comes from a small, transparent booking fee charged to the driver's prepaid wallet, only on completed Booking rides. Shared rides are always free.

The "reward" is credited back to the driver's wallet on completion, softening the net cost and rewarding activity. Example: on a ₱60 booking, the driver collects ₱60 cash from the passenger, pays ₱8 platform fee from their wallet, and receives ₱2 back — a net platform revenue of ₱6 per ride.

4.2 Prepaid Wallet System

  • Top-up based: Drivers maintain a prepaid balance; fees are auto-deducted on ride completion. Collection is guaranteed — there is no debt risk.
  • Welcome bonus: New drivers receive ₱15 free starting balance to onboard smoothly.
  • Minimum to accept: A driver needs at least ₱10 balance to accept a Booking ride (enough to always cover the fee).
  • Immutable ledger: Every fee, reward, top-up, referral bonus, and adjustment is recorded as a permanent transaction record for full auditability.
  • Admin top-up: Administrators can credit driver wallets (e.g. after GCash payment) directly from the admin dashboard.

4.3 Driver Referral Program

Each driver gets a unique referral code. When a referred driver completes their first booking ride, the referrer earns a ₱10 wallet bonus. The "first completed ride" trigger ensures bonuses only reward genuinely active drivers — it is resistant to fake-signup farming.

5. Real-Time Location & Live Tracking

5.1 Redis Live-Location Layer

High-frequency driver GPS updates are stored in Redis (in-memory) rather than the main database. This is the same "hot location layer" pattern used by major ride-hailing platforms.

  • Speed: Location reads/writes hit memory, not disk — sub-millisecond, and they never burden the primary database.
  • Self-cleaning: Each location has a 60-second TTL (time-to-live). A driver who loses signal or goes offline automatically disappears from the live map — no cleanup job needed.
  • Powers: the admin live map, passenger live-tracking of their assigned driver, and dispatch freshness checks.

5.2 Smooth Location Querying & Bandwidth Efficiency

  • Movement-gated updates: The driver app only sends location when the driver actually moves (distance filter), so a parked driver doesn't flood the server with redundant pings.
  • Adaptive rate-limiting: Network pushes are throttled (more frequent during an active ride when a passenger is watching, less frequent when idle) — balancing freshness against battery and data usage.
  • Heartbeat: A lightweight 30-second "still here" ping keeps a stationary driver marked online without continuous GPS streaming.

5.3 In-App Navigation

  • Heading-aware markers: The driver and the tracked vehicle are shown as directional arrows that rotate to the real travel heading — a true turn-by-turn navigation feel.
  • Smooth follow camera: The map glides with the driver continuously (no jarring snapping), and can be freely panned, automatically resuming follow.
  • Auto-framing: When a request arrives, the map auto-zooms to show driver, pickup, and destination together so the driver can judge the trip at a glance.
  • Google Maps handoff: One tap launches full external Google Maps turn-by-turn navigation to the pickup or drop-off.

5.4 Background Operation (Always-Online)

The driver app runs a proper Android foreground service, so a driver stays online and keeps reporting location even when the app is minimized or the screen is locked — exactly like Grab/Uber driver apps. The service starts when the driver goes online and stops when they go offline, preserving battery.

7. Security & Reliability

  • Encrypted credentials: passwords hashed with bcrypt; auth tokens stored in the Android Keystore.
  • Live status enforcement: suspended/banned drivers are blocked on every request, not just at login.
  • Rate limiting: brute-force login protection and Google Maps cost protection via Redis counters.
  • No leaked internals: network errors shown to users are sanitized — the backend domain is never exposed.
  • Input validation: every API request is schema-validated (Zod) before processing.
  • Resilient FCM: push tokens self-heal on every app launch, so notifications keep working across reinstalls and updates.
  • Always-on hosting: the backend runs under a process manager with auto-restart, behind Nginx with HTTPS.
